## 长对话性能：消息节点 v3 分片索引

**问题**：旧消息表按整数组读写。对话变长后，首次加载、发送、状态书计数和蒸馏都会读取完整历史，即使高性能机器也会明显卡顿。

**核心方案**：
- `tavern_message_nodes` 升级到 `__message_nodes_v3__`
- 每条消息独立存储为 `node:{id}`
- 每个会话维护 `conversation_index:{conversationId}`
- 索引元数据包含 `id/timestamp/role/isArchived/hasScribeUpdate`
- 首次访问自动从旧整表或 v2 分片重建 v3 索引
- 迁移失败会释放缓存 Promise，后续操作可重试

**新增接口**：
- `getMessageNodesPageByConversation()`：80 条分页读取
- `queryMessageNodesByConversation()`：按角色、归档状态、顺序和数量有限查询
- `countMessageNodesByConversation()`：仅扫描轻量索引计数
- `getMessageNodeMetadataByConversation()`：蒸馏规划只读元数据，不加载全文
- `replaceAllMessageNodes()`：备份恢复时重建全部分片与索引

**复合游标**：分页游标使用 `{ timestamp, id }`，同毫秒消息不会被跳过。

**发送链路**：普通聊天只读取最近上下文、最新状态书和必要蒸馏节点；完整历史读取保留给分支、导出、重试等明确操作。

## 完整轮次蒸馏系统重构

**原问题**：
- 手动蒸馏把触发阈值当作最大批次，没有最低启动条件
- 默认保留 3 条后再聊一轮，手动点击会只蒸馏 2 条消息
- 自动蒸馏按消息条数切割，可能产生奇数批次，拆开 user/assistant
- 自动与手动没有执行互斥，可能重复总结同一批来源节点
- 空响应会写入“蒸馏失败”并归档原对话
- 先写结晶、后归档的两步提交可能留下半完成状态

**统一规划器**：新增 `src/utils/distillation.ts`。

`planDistillation(candidates, triggerThreshold, retainRecentRounds)` 行为：
- 按时间与 ID 稳定排序
- 以 user 节点为轮次起点
- 一轮包含该 user 与下一条 user 之前的所有 charA/charB 回复
- 只处理包含角色回复的完整轮次
- 自动与手动均选择最旧的 `triggerThreshold` 个完整轮次
- 最近 `retainRecentRounds` 个完整轮次整体保留
- 不足阈值直接返回空计划，不调用模型

**默认行为**：阈值 10 轮、保留最近 3 轮，即至少积累 13 个完整未归档轮次才会产生下一批结晶。

**并发和提交保护**：
- `useDistillation` 使用同步 `inFlightRef`，同一页面只允许一个蒸馏任务
- `commitDistillationBatch(sourceIds, distilledNode)` 在 message store 写锁内重新读取来源
- 来源节点已归档时拒绝重复提交
- 同一锁内完成来源归档、索引更新和结晶写入
- 写入失败时恢复来源节点和原索引，并移除半写入结晶

**输出保护**：
- 兼容 `content/text/reasoning_content`
- 统一使用 `stripReasoningBlocks()` 清理推理块
- 清理后为空则抛错，原对话保持未归档
- `{dialogue}` 占位符现在会实际替换；无占位符的旧自定义提示词仍兼容

**累计记忆**：
- 新结晶带 `distillationMeta`：来源节点、真实起止轮次、`cumulative=true`
- 新摘要会把上一份累计记忆与本批变化合并
- 上下文只注入最新累计结晶，避免最近多份累计摘要重复占用 Token
- 没有元数据的旧版结晶仍按 `maxDistilledNodes` 兼容
- 默认结晶模板升级为 `第{start}轮-第{end}轮`

## 可编辑记忆回廊

**实现**：

- `src/components/chat/ChatInput.tsx`

## 缓存世界书 JSON 与类型边界加固

**问题**：AI 返回 `null`、对象键、非法 operation 或非有限 priority 时可能崩溃；旧启发式还会把名称相似或恰好 10 条的手动世界书误判成缓存书。

**修复**：
- 仅显式 `WorldBook.kind === 'cache'` 才视为缓存世界书
- 状态书写入前再次校验目标确实是 cache 类型
- `mergeCacheWorldBookEntries()` 接受 unknown 并逐项验证
- keys 只接受字符串/数字，最多 6 个
- value 必须是非空字符串并限制长度
- priority 必须为有限数字并 clamp 到 1-10
- operation 仅接受 `upsert/delete`
- A 世界书已有关键词在代码层阻止重复写入缓存书
- 手动世界书关键词注入提示词限制数量与字符数
- 删除世界书时清理角色卡的 `worldBookId/cacheWorldBookId`
- 普通与缓存世界书 JSON 导入均防御 `null` 和非对象条目
